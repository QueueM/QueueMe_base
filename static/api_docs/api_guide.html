QueueMe API Guide

Welcome to the QueueMe API Guide. This comprehensive documentation will help you integrate QueueMe's powerful queue management and booking features into your applications. The QueueMe API follows REST principles, uses JSON for data exchange, and provides standardized responses for a seamless integration experience.

Whether you're building a custom booking system, integrating with an existing application, or creating a mobile app, this guide will walk you through all the essential concepts and operations.

Note: This guide complements the interactive API documentation available through our Swagger UI and ReDoc interfaces. For detailed endpoint specifications, request/response examples, and live testing, please refer to those resources.

Authentication

QueueMe API uses JWT (JSON Web Tokens) for authentication. All API requests must include an Authorization header with a valid token.

Obtaining Access Tokens

To obtain an access token, make a POST request to the authentication endpoint with valid credentials:

curl 
curl -X POST "https://api.queueme.net/api/v1/auth/token" \
     -H "Content-Type: application/json" \
     -d '{
       "username": "your_username",
       "password": "your_password"
     }'

The response will include an access token and a refresh token:

json 
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 86400
}

Using Access Tokens

Include the access token in the Authorization header of all API requests:

curl 
curl -X GET "https://api.queueme.net/api/v1/shops" \
     -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."

Token Expiration and Refresh

Access tokens expire after 24 hours. When a token expires, use the refresh token to obtain a new access token without requiring the user to authenticate again:

curl 
curl -X POST "https://api.queueme.net/api/v1/auth/token/refresh" \
     -H "Content-Type: application/json" \
     -d '{
       "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
     }'
Warning: Keep your tokens secure and never expose them in client-side code. Use a server component to handle sensitive operations and token management.

Making Requests

The QueueMe API accepts requests in standard REST format over HTTPS. All data is sent and received in JSON format.

API Base URL

All API requests should be directed to the base URL: 

https://api.queueme.net/api/v1/

HTTP Methods

The API uses standard HTTP methods to indicate the desired action:

Method Description Example
GET Retrieve resources GET /shops to list all shops
POST Create a new resource POST /bookings to create a new booking
PUT Update a resource (full update) PUT /shops/123 to update a shop
PATCH Update a resource (partial update) PATCH /shops/123 to update specific fields
DELETE Delete a resource DELETE /bookings/456 to cancel a booking

Request Headers

Include the following headers with your requests:

Header Description Example
Authorization Bearer token for authentication Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Content-Type Media type of the request body Content-Type: application/json
Accept Preferred response format Accept: application/json

Request Body

For POST, PUT, and PATCH requests, include the request body as a JSON object:

curl 
curl -X POST "https://api.queueme.net/api/v1/bookings" \
     -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
     -H "Content-Type: application/json" \
     -d '{
       "service_id": 123,
       "specialist_id": 456,
       "date": "2025-06-15",
       "time_slot": "14:30",
       "customer_id": 789
     }'

Query Parameters

Use query parameters to filter, sort, paginate, or specify response options:

curl 
curl -X GET "https://api.queueme.net/api/v1/services?shop_id=123&category=haircut&limit=10&offset=0" \
     -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
Tip: Use the include parameter to expand related resources in the response, reducing the number of API calls needed. For example, ?include=specialist,shop will include full details of the specialist and shop in the service response.

Handling Responses

All API responses are returned in JSON format with consistent structure for easy parsing and handling.

Success Responses

Successful responses include an HTTP status code in the 2XX range and a JSON payload containing the requested data:

json 
{
  "status": "success",
  "data": {
    "id": 123,
    "name": "Central Salon",
    "address": "123 Main Street",
    "city": "Cityville",
    "phone": "+1234567890",
    "email": "contact@centralsalon.com",
    "created_at": "2025-01-15T10:30:00Z",
    "updated_at": "2025-05-20T14:22:10Z"
  }
}

For list endpoints, the response includes pagination information:

json 
{
  "status": "success",
  "data": [
    {
      "id": 123,
      "name": "Central Salon",
      ...
    },
    {
      "id": 124,
      "name": "Downtown Barbershop",
      ...
    }
  ],
  "pagination": {
    "total": 42,
    "count": 2,
    "per_page": 10,
    "current_page": 1,
    "total_pages": 5,
    "links": {
      "next": "https://api.queueme.net/api/v1/shops?page=2",
      "prev": null
    }
  }
}

Error Responses

Error responses include an HTTP status code in the 4XX or 5XX range and a JSON payload with error details:

json 
{
  "status": "error",
  "error": {
    "code": "validation_error",
    "message": "The request was invalid",
    "details": [
      {
        "field": "date",
        "message": "Date must be in the future"
      },
      {
        "field": "time_slot",
        "message": "Time slot must be in 30-minute increments"
      }
    ]
  }
}

Common HTTP status codes in error responses:

  • 400 Bad Request - The request was malformed or invalid
  • 401 Unauthorized - Missing or invalid authentication
  • 403 Forbidden - Authentication successful but insufficient permissions
  • 404 Not Found - The requested resource does not exist
  • 409 Conflict - The request could not be completed due to a conflict
  • 422 Unprocessable Entity - Validation errors
  • 429 Too Many Requests - Rate limit exceeded
  • 500 Internal Server Error - Server-side error
